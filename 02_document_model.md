# 02 — The Document Model

## Storage vs. Compute: A Historical Shift

**1970s context:** Storage was the most expensive compute component. SQL databases were designed to *minimize storage* through **normalization** — splitting data into multiple tables and using keys to reference shared values, avoiding redundancy.

**Modern context:** Storage is cheap; compute is the bottleneck. We now have more varied data from more sources, and the cost of re-joining normalized data (computationally expensive operations) outweighs the cost of storing some redundancy.

> NoSQL databases were created to prioritize **flexibility and compute efficiency** over storage minimization.

### Normalization Example (SQL)
Instead of repeating a vendor's address on every order record, SQL stores the vendor once and references it via a `VendorID`. Querying requires joining the orders table with the vendors table — an expensive operation at scale.

---

## Polymorphic Data

MongoDB's document model handles **polymorphic data** — records that don't all share the same fields or structure. You can combine different types of data in a single collection without breaking the schema.

When you add a new feature that requires a new field, you add it only to relevant documents. Existing documents are unaffected. No table-wide migration required.

---

## MongoDB's Hierarchy

```
Cluster
  └── Node
        └── Database
              └── Collection
                    └── Document
```

- **Document** — the atomic unit of data (equivalent to a row, but flexible)
- **Collection** — a group of documents (equivalent to a table, but schema-optional)
- **Database** — a namespace containing collections
- **Node** — a single server/machine
- **Cluster** — a group of nodes working together

---

## The Golden Rule

> **Data that is accessed together should be stored together.**

This drives schema design in MongoDB. Unlike SQL where you normalize to save storage, in MongoDB you *embed* related data in a single document to avoid joins at read time.

### Car Analogy
- **SQL:** Disassemble the car to store parts in separate containers. Efficient storage, but must reassemble on every read.
- **MongoDB:** Store the car intact. Faster reads, and new components don't require creating new containers.

## Document Syntax (JSON/BSON)

```json
{
  "_id": ObjectId("5f4f7fef2d4b45b7f11b6d7a"),
  "user_id": "Sean",
  "age": 29,
  "status": "A"
}
```

### Syntax Rules
| Element | Rule |
|---|---|
| Document boundaries | Wrapped in `{ }` |
| Field-value separator | Colon `:` |
| String values | Wrapped in `" "` |
| Numeric values | No quotes |
| Field-value pairs | Separated by commas `,` |

### The `_id` Field
- Serves as the **primary key**
- Must be **unique** within the collection
- Is **immutable** (cannot be changed after creation)
- Can be any type except array
- Auto-generated by default (as an `ObjectId`), but can be set manually

---

## Collections

- A collection is a group of documents, typically with similar contents
- Common fields between documents are conventional but not required unless schema validation is active
- MongoDB supports **schema validation** to enforce field rules when needed

> **Misconception to avoid:** "Non-relational" does not mean relationships don't exist in the data. Relationship design is a *core* part of MongoDB schema design — it just works differently than SQL foreign keys.

---

## Key Takeaways

- The shift from storage-constrained to compute-constrained environments is why NoSQL exists.
- MongoDB's document model is optimized for reading data in its natural, assembled form.
- Hierarchy: Document → Collection → Database → Node → Cluster
- The golden rule: store data together if it will be accessed together.
- `_id` is always the primary key; unique, immutable, auto-generated by default.
